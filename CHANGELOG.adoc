= Changelog

== Unreleased

* New channels: Matrix (`matrix://<access-token>@<homeserver>/<room-id>`, Client-Server API),
  Mastodon (`mastodon://<access-token>@<instance-host>`), and Bluesky
  (`bluesky://<identifier>:<app-password>[@<pds-host>]`, AT Protocol — creates a session then
  posts).
* `ExpiringFilter` accepts an optional `Clock` so expiry can be driven deterministically
  (the existing constructor still uses the system clock).
* Channel discovery API (`ChannelCatalog`, via `ChannelCatalog.standard()`): enumerate every
  channel and its fields (`catalog()`/`describe()`), build a valid URL from field values
  (`buildUrl()`), validate required fields (`validate()`), decompose an existing URL back into
  fields for editing with secrets masked (`parse()`), and redact a URL for display
  (`redact()`). Lets a consumer render a "pick a channel → fill the fields" UI data-driven —
  every channel notify4j adds appears automatically. notify4j owns URL assembly, so
  `buildUrl` output round-trips through the parser (pinned by a per-scheme test).

== 1.0.0 (2026-06-28)

*API stability:* this is the 1.0 line. The public API — the `Notifier`,
`NotificationAdapter`, `NotificationFilter`, and `NotificationMetrics` SPIs; the
`Notifications` / `NotificationsFactory` / `NotificationsConfig` facade; the value types
(`Severity`, `HttpClientConfig`); the decorators (`AsyncNotifier`, `CompositeNotifier`,
`FilteringNotifier`, `RemindingNotifier`, `LoggingNotifier`); the `Abstract*` extension
points; and the `notify4j.*` properties and the channel-URL grammar — follows semantic
versioning within 1.x. The `org.alexmond.notify4j.internal` package (the concrete channel
notifiers) is *not* public API: build channels via URLs, not by constructing those classes.
Both jars carry an `Automatic-Module-Name` (`org.alexmond.notify4j` / `…spring`).

=== Breaking

* The `Notifications` / `NotificationsFactory` telescoping constructors are replaced by an
  immutable, builder-built `NotificationsConfig` (ignore-changes, log sink, HTTP, executor,
  metrics). Use `new Notifications<>(urls, adapter, config)` /
  `new NotificationsFactory<>(adapter, config)`; `Notifications.setMetrics(...)` is removed
  (set `.metrics(...)` on the config).
* The 18 concrete channel notifiers moved to `org.alexmond.notify4j.internal` and are no
  longer public API. Construct channels via URLs (the supported path), not by instantiating
  `SlackNotifier` etc. directly.

=== Added

* Optional `title` and `severity` on `NotificationAdapter` (default methods, so existing
  adapters are unaffected). Channels with a native notion map them — PagerDuty severity (was
  hard-coded `error`), OpsGenie priority, ntfy/Gotify/Pushover priority + title, and the
  email subject. A new `Severity` enum (`DEFAULT`, `INFO`, `WARNING`, `ERROR`, `CRITICAL`);
  `DEFAULT` preserves the previous wire format.
* `notify4j.async.queue-capacity` (default 1000) and `notify4j.async.rejection-policy`
  (`DROP` / `CALLER_RUNS`) bound the delivery queue so a backlog can't OOM the host; drops
  are logged and recorded via `NotificationMetrics.recordDropped`.
* Reminders are configurable from the starter (`notify4j.reminders.*`): re-notify every
  channel for an entity stuck in a configured status until it resolves. The facade forgets
  the entity's transition state before each reminder so the repeat isn't suppressed as a
  non-change; the scheduler is started/stopped with the facade lifecycle.
* Non-blocking retry: with async delivery, transient-failure backoff is scheduled
  off-thread (via the async HTTP client) so it never ties up the delivery pool
  (`notify4j.http` retry is unchanged for synchronous use).
* The WhatsApp Graph API version is overridable per channel (`whatsapp://…?version=v22.0`).
* IDE configuration metadata for the `notify4j.*` properties (autocomplete + hover docs).
* `Notifications` is `AutoCloseable`: `close()` closes resource-holding channels (e.g. a
  `RemindingNotifier`, also `AutoCloseable`). The config executor is caller-owned.
* `SECURITY.md` with a private disclosure channel and the security/trust model.

=== Changed / fixed

* *Security:* URL-parse errors (thrown at startup) no longer leak the secret-bearing URL —
  messages keep only the scheme. The shared `HttpClient` pins `Redirect.NEVER`, and a
  cleartext `+http` on a credential-bearing channel logs a warning. The email subject is
  CRLF-sanitised.
* The starter's async executor is a `ThreadPoolTaskExecutor`, shut down gracefully on
  context close (waits for in-flight deliveries, then interrupts).
* The channel-URL grammar is documented as a frozen contract (transport suffix, `?tags=`,
  credential default hosts, the zulip/twilio hand-parse rules).

== 0.7.0 (2026-06-25)

* *Security:* channel URLs carry secrets (Telegram/Gotify tokens in the URL, opaque
  Slack/Discord webhook URLs), so they are no longer logged on delivery failure — exception
  messages use a redacted scheme+host form, and the failing event object is no longer logged.
* *Robustness:* a misbehaving adapter/filter can no longer break the caller (the
  transition guard now runs inside the error-isolation try); `TransitionFilter` is atomic and
  null-safe; the runtime-mute check no longer prunes on the delivery hot path; retry backoff is
  clamped; and the hand-parsed channel URLs strip a stray query before splitting.

== 0.6.0 (2026-06-24)

* New channel: Pushover (`pushover://<app-token>/<user-key>`) — hosted mobile/desktop push,
  the form-encoded counterpart to the self-hosted ntfy/Gotify channels.
* New channels: SMS via Twilio (`twilio://<sid>:<token>@<from>/<to>`), Signal
  (`signal://<bridge-host>/<from>/<to>`, via a signal-cli REST bridge), and WhatsApp
  (`whatsapp://<token>@<phone-number-id>/<to>`, Meta Cloud API).
* New channels: Zulip (`zulip://<bot-email>:<api-key>@<host>/<stream>/<topic>`) and Pushbullet
  (`pushbullet://<access-token>`).
* Docs: how to wire async/retry/metrics onto a per-tenant `NotificationsFactory` without the
  starter (the programmatic multi-tenant path).

== 0.5.0 (2026-06-23)

* Optional Micrometer metrics: per-channel `notify4j.notifications` counters tagged
  `channel` + `outcome` (sent/failed/suppressed), so operators can confirm delivery. The core
  stays dependency-free via a `NotificationMetrics` SPI; the starter wires a Micrometer-backed
  implementation automatically when a `MeterRegistry` bean is present (e.g. with Actuator).
* `AbstractHttpNotifier` gains an overridable `contentType()` and a `formEncode(...)` helper,
  enabling non-JSON, form-encoded channels (e.g. Pushover, Twilio) to drop in.

== 0.4.0 (2026-06-23)

* *Teams:* now posts a proper MessageCard instead of a bare `{"text":…}`, so it works with
  Power Automate *Workflows* webhooks — the legacy Office 365 connector webhooks were retired
  by Microsoft in May 2026. Point the `teams://` URL at a Workflows webhook.
* New channels: Mattermost (`mattermost://`), Rocket.Chat (`rocketchat://`), Google Chat
  (`googlechat://`), and Gotify (`gotify://`) — completing parity with shoutrrr's webhook-style
  services.

== 0.3.0 (2026-06-22)

* Asynchronous, non-blocking delivery: the starter wraps each channel in an `AsyncNotifier`
  and dispatches on a shared pool, so `send` returns immediately and a slow channel can't
  block the caller or its siblings. *On by default*; disable with `notify4j.async.enabled=false`
  (configurable pool via `notify4j.async.pool-size`). The core stays synchronous unless an
  `Executor` is supplied to `Notifications`/`NotificationsFactory`.
* Transient-failure retry for the webhook-style channels: HTTP 5xx/429 and `IOException` are
  retried with exponential backoff; other 4xx fail fast. *On by default* in the starter
  (`notify4j.http.max-attempts=3`, `notify4j.http.retry-backoff=500ms`); core programmatic
  default remains single-attempt.

== 0.2.0 (2026-06-22)

* Configurable HTTP timeouts (`notify4j.http.connect-timeout` / `read-timeout`) for the
  webhook-style channels, which now share a single `HttpClient` instead of one per channel.
* *Breaking:* the webhook-style channel notifier constructors (`SlackNotifier`,
  `TeamsNotifier`, `DiscordNotifier`, `WebhookNotifier`, `TelegramNotifier`, `NtfyNotifier`,
  `PagerDutyNotifier`, `OpsGenieNotifier`) now take an `HttpClientConfig`. The
  `Notifications` / `NotificationsFactory` facade API is unchanged.
* Email subject prefix now defaults to `[notify4j]` (was a leftover `[builder]` literal).
* PagerDuty payload `source` is now `notify4j` (was a leftover `builder` literal).
* Upgraded to Spring Boot 4.0.7.
* Expanded documentation: getting-started, configuration (full channel-URL reference),
  architecture, and contributing pages; sample app ships an example `application.yml`.
* Per-channel payload-shape tests pinning the exact wire format of every channel.

== 0.1.0

* Initial release. Multi-module Maven layout (`notify4j-parent` + `-bom` + `-core` +
  `-spring-boot-starter` + `-sample`), Java 17, Spring Boot 4.0.x, published to Maven Central.
* Spring-free core: `Notifier<E>` SPI, `NotificationAdapter<E>`, `Notifications` facade,
  `NotificationsFactory`, and the Apprise/shoutrrr-style `NotifierUrlParser`.
* Channels: Slack, Teams, Discord, Telegram, ntfy, generic webhook, PagerDuty, OpsGenie, and
  email (via the Spring Boot starter).
* Status-transition filtering, tag-based routing, runtime mute, and reminders.
* Quality gates: spring-javaformat, Checkstyle, PMD, and JaCoCo coverage.
