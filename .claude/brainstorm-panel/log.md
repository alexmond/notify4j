- Proposed: API Steward, Reliability/SRE, Security, Consumer DX, Competitive/Product, Release Manager (skeptic). User added: none. User removed: none.
- Style: swarm → director-led (Release Manager converges; API Steward veto on freeze). Fit well for a scope/cut-line decision.
- Recurring (worth seating again on this repo): **API Steward** and **Release Manager** are essential for any release/API-freeze target. **Security** + **Reliability** each surfaced a host-impacting issue the prior 4-lens review missed (SSRF; unbounded async queue → OOM) — keep both for anything touching delivery or URLs.
- Key cross-lens findings the earlier review missed: (1) async pool has an **unbounded queue → can OOM the host**; (2) **SSRF** via user-supplied channel hosts (multi-tenant); (3) **no title/severity** in the SPI (`NotificationAdapter` is id/status/message) — channels flatten everything, PagerDuty hardcodes `severity:"error"`; (4) **secret leak in URL-*parse* exceptions** (regression of 0.7.0 log redaction, on the startup path); (5) **no IDE config metadata** (`additional-spring-configuration-metadata.json`); (6) **no SECURITY.md**; (7) `NotificationMetrics`/SPI need `default` methods to stay evolvable past the freeze.
- Adjudication rule that worked: "blocks 1.0" = *irreversible after the freeze* (API Steward) **or** *credibility/safety failure in a published 1.0* (Release Manager). Anything addable compatibly later (new property, `default` SPI method, new channel) is 1.0.x/1.1.
- Note: a 1.0 here is overwhelmingly an **API-surface promise**, not a feature ceiling — the only strictly-irreversible work is the public/internal boundary (#37) and the URL grammar (#38).

## 2026-06-29 — channel catalog / discovery API (#70) (design a new frozen public API)
- Proposed: API Steward, Consumer DX, Parser/Internals, Security, Scope Skeptic. User added/removed: none.
- Style: swarm → director-led (API Steward converges, Skeptic guards scope). Fit well for an API-design + scope-cut target.
- Recurring (now 2nd API target): **API Steward + Scope Skeptic + Security** are the load-bearing trio for any public-API/freeze design here — seat them by default. A **Parser/Internals** lens is essential whenever the feature touches the URL grammar.
- Key cross-lens findings: (1) the URL is NOT a flat field bag — 4+ bespoke grammars (zulip last-@, twilio/whatsapp first-@ with `+`, matrix `:` room-id, credential-only default-host) → `buildUrl` must OWN assembly; consumers must never concatenate. (2) `credentialBearing` must be DERIVED from `anyMatch(field.secret)`, NOT reused from `CREDENTIAL_SCHEMES` (which excludes slack/webhook whose whole URL is the secret) — Security's top catch. (3) Anti-drift = a registry-wide `parse(buildUrl(...))` round-trip test with *nasty* fixtures; default-host elision (pagerduty/opsgenie/pushbullet/pushover is many-to-one) is the sneaky one a clean test misses. (4) Expose the existing redactor publicly (`redact(url)`) — biggest cheap security win. (5) Don't rewrite the parser onto descriptors (frozen 1.0 grammar, weak type-only tests) — add a parallel descriptor registry + round-trip test instead.
- Scope verdict: ship enumerate + describe + buildUrl + minimal field-keyed validate + redact; DEFER parse-back-to-fields (edit screens), category/displayName/help/docUrl, ENUM/NUMBER/BOOL field types, attributes extension map. Closed `FieldType` enum is a breaking-change generator → keep tiny + document "consumers must default"; split `secret` out as its own boolean.
- Note: there is NO field getter on notifiers today, so round-trip tests assert at the URL-string + notifier-type level, not reconstructed endpoints.

## 2026-07-02 — one-shot imperative send (#81) (choose-between-options + design new frozen API)
- Proposed: API Steward, Consumer DX, Reliability/lifecycle, Scope Skeptic, Security. User added/removed: none.
- Style: swarm → director-led (API Steward converges, Scope Skeptic guards scope). Fit well — 3rd API target running this exact roster/style.
- Recurring (now 3rd API target): **API Steward + Scope Skeptic + Security** trio confirmed load-bearing — graduate to the repo's default API-target panel. **Consumer DX** earns its seat whenever the issue is filed from consumer pain; **Reliability** whenever delivery lifecycle is touched. This roster is now the standard "new public API on notify4j" panel.
- Key cross-lens findings: (1) **`TransitionFilter.allow(null, status)` returns true and stores NOTHING** (TransitionFilter.java:32-35) → map the one-shot adapter's `id → null`; beats `identityHashCode` / per-message UUID because a reused handle never grows the lastStatus map. Reliability's code-grounded top catch. (2) One-shot MUST be **sync + blocking** (executor null + `nonBlockingRetry=false`) or a short-lived pipeline JVM exits before the POST on daemon threads. (3) `includeLog=false` on the one-shot config so it's exactly-N deliveries, no double-log. (4) Security: **`redact()` LEAKS on credential-in-authority schemes** (pagerduty/opsgenie/pushbullet/pushover — the secret parses as the URI host, redact returns it verbatim) → a latent bug in the just-shipped `redact()`, and the reason one-shot labels must use scheme-only `NotifierUrlParser.safe()`, not `redact()`. (5) keep retry + redaction + never-throw; bypass transition/mute/reminder/async/tag-routing.
- Adjudications (director): **place the static on the existing facade** (`Notifications.sendOnce`, not a rival `Notify` noun) — coherence over a second entry point. **`Message` adapter stays package-private** (exposing a bogus-id/status shim freezes it forever; nothing needs it once `sendOnce` exists) — Skeptic over Steward. **Drop `Message.tags`** (a one-shot fires exactly the URLs you pass it; per-message route tags are redundant and `?tags=` already owns routing; record components can't widen) — defer to a `sendOnce(urls,msg,routeTags)` overload if ever needed. **Return a minimal `SendResult(attempted,sent,failed)`** (counts only, NO per-channel labels) — threads Reliability's "CLI needs an exit code" against Skeptic's "no per-channel result design" and Security's leak surface (no URLs in the result at all).
- Non-goals stated for 1.1.0: fluent `Notify.to(...)` builder, async/`CompletableFuture` one-shot, reusable amortized `ofMessages` handle, Spring one-shot bean, scheduled/repeat sends, attachments, a named-channel registry (stays consumer-side per the issue).
- Note: the 3-arg tuning overload should take **`HttpClientConfig`** (timeouts/retry only), NOT a full `NotificationsConfig` — a config carrying an executor would silently make `sendOnce` fire-and-return-before-delivery (the async footgun both API Steward and Reliability flagged).
